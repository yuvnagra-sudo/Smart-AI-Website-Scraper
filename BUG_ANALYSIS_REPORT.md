# Bug Analysis Report ‚Äî Smart-AI-Website-Scraper
**Date:** March 6, 2026  
**Repository:** `yuvnagra-sudo/Smart-AI-Website-Scraper`  
**Analyst:** Manus AI  
**Scope:** Three reported production issues with the Agent/Custom Scraper mode

---

## Executive Summary

After a full code review across `server/agentScraper.ts`, `server/routers.ts`, `server/worker.ts`, `server/_core/jobCancellation.ts`, and the React frontend (`client/src/pages/Dashboard.tsx`), all three reported issues have been traced to specific root causes. Two of the three bugs are **architectural** ‚Äî meaning they cannot be fixed with a small patch and require a targeted structural change. The third is a combination of a **prompt engineering constraint** and a **missing data-flow validation step**. None of the three issues require a full overhaul of the codebase; the existing architecture is sound and the fixes are surgical.

## Issue 1: AI Prompt Does Not Produce Flexible or Filled Output Columns

### Symptom

When a user enters a natural-language description of what they want to extract, the generated sections either do not match what was asked, or the output Excel file has columns that are mostly empty even though the data is present on the scraped websites.

### Root Causes

**Root Cause 1A ‚Äî The `generateExtractionPlan` system prompt artificially limits sections to 3‚Äì8.**

The server-side prompt that generates the extraction plan instructs the LLM: *"3-8 sections total."* This is a hard constraint baked into the system message. If a user asks for 12 fields (e.g., "I want name, email, phone, address, founding year, employee count, revenue, tech stack, CEO name, LinkedIn, Twitter, and industry"), the LLM will silently merge or drop fields to comply with the 8-section cap. The user sees fewer columns than they asked for, with no explanation.

```
// server/routers.ts ‚Äî generateExtractionPlan system message
"Rules for sections:
- 3-8 sections total"          ‚Üê hard cap, no user override
```

The post-processing code does allow up to 15 sections (`.slice(0, 15)`), but the LLM never generates more than 8 because the system prompt tells it not to.

**Root Cause 1B ‚Äî The `extractProfileFields` function truncates page content to 25,000 characters.**

Inside `agentScraper.ts`, the LLM extraction call passes only the first 25,000 characters of the scraped page:

```typescript
// server/agentScraper.ts ‚Äî extractProfileFields
const userMsg = `...
Page content:
${content.substring(0, 25000)}   ‚Üê hard truncation
...`;
```

For content-rich websites (e-commerce, SaaS product pages, large company sites), this truncation silently discards the portion of the page that may contain the exact fields the user requested. The LLM then returns empty strings for those fields ‚Äî not because the data does not exist, but because it was cut off before the LLM could see it.

**Root Cause 1C ‚Äî The `systemPrompt` generated by the AI plan is not validated against the actual `sections` keys.**

The `generateExtractionPlan` mutation produces both a `sections` array and a `systemPrompt` string. The `systemPrompt` is a free-text instruction block that the LLM uses during extraction. However, there is no check that the field names mentioned in the `systemPrompt` match the `key` values in the `sections` array. If the LLM generates a `systemPrompt` that says "extract the CEO's name" but names the field `ceo_full_name`, while the `sections` array has `key: "ceo_name"`, the extraction result will be returned under `ceo_full_name` ‚Äî which does not match any column header ‚Äî and the column `ceo_name` will be empty.

**Root Cause 1D ‚Äî The `mergeResults` function uses a "prefer non-empty" strategy but does not handle partial overwrites.**

When multiple sub-pages are scraped and merged, `mergeResults` keeps the first non-empty value for each key. If the homepage returns a partial value (e.g., a truncated description) and a sub-page returns the full value, the partial value from the homepage is kept and the better value from the sub-page is discarded. This causes columns to appear filled but with low-quality data.

### Recommended Fixes

| Fix | Location | Effort |
|---|---|---|
| Raise the section cap to 15 and expose a UI slider for "number of columns" | `routers.ts` system prompt + `Dashboard.tsx` | Low |
| Implement chunked extraction ‚Äî split content into 25K windows, extract each, then merge | `agentScraper.ts` ‚Äî `extractProfileFields` | Medium |
| Add a post-generation validation step that checks `systemPrompt` references match `sections` keys | `routers.ts` ‚Äî `generateExtractionPlan` | Low |
| Change `mergeResults` to prefer the **longer** non-empty value, not just the first | `agentScraper.ts` ‚Äî `mergeResults` | Low |

## Issue 2: Job Immediately Shows as Finished in the UI While Still Running in Railway Logs

### Symptom

After clicking "Start Job," the UI briefly shows "processing" and then flips to "completed" within seconds, even though the Railway worker logs show the job is still actively scraping URLs. The output file either does not exist or contains only a few rows.

### Root Causes

**Root Cause 2A ‚Äî The web server and the worker are two separate OS processes that do NOT share in-memory state.**

This is the primary architectural cause. The project runs two Node.js processes inside the Railway container: `node dist/index.js` (the web server) and `node dist/worker.js` (the background worker). They communicate exclusively through the MySQL database.

The `cancelledJobs` Set in `_core/jobCancellation.ts` is an **in-memory singleton**. When the web server's `cancelJob` mutation writes `status = "cancelled"` to the database, it does **not** call `markJobCancelled()` ‚Äî it only updates the DB. The worker's `checkForCancellation` timer polls the DB every 5 seconds and calls `markJobCancelled()` in its own process memory. This is correct and intentional.

However, the same process-isolation problem causes Issue 2: if the web server's `resumeJob` mutation calls `processEnrichmentJob()` directly (which it does ‚Äî see line 394 of `routers.ts`), the job runs inside the **web server process**, not the worker process. The web server has no heartbeat timer, no cancellation check timer, and no `keepAlive` connection manager running for that job. When the web server's HTTP request times out or the connection drops, the job function is abandoned mid-run ‚Äî but the DB still shows `status = "processing"`. On the next worker poll cycle, the worker sees a stale heartbeat and resets the job to `pending`, picks it up, and starts from the beginning. This creates the illusion of "immediately finishing" ‚Äî the UI sees the status bounce from `processing` ‚Üí `pending` ‚Üí `processing` again, which can look like a fast completion.

**Root Cause 2B ‚Äî `processAgentJob` is missing `keepAlive.start()`.**

In `processEnrichmentJob` (the VC mode), the database keep-alive connection manager is started immediately:

```typescript
// processEnrichmentJob ‚Äî CORRECT
const keepAlive = new ConnectionKeepAlive();
keepAlive.start();   ‚Üê present
```

In `processAgentJob` (the custom AI scraper mode), `keepAlive` is instantiated but **never started**:

```typescript
// processAgentJob ‚Äî BUG
const keepAlive = new ConnectionKeepAlive();
// keepAlive.start() is MISSING
try {
  ...
} finally {
  keepAlive.stop();  ‚Üê stop is called on something that was never started
}
```

For long-running agent jobs (many URLs, many LLM calls), the MySQL connection will time out after the default idle period (~8 hours on most providers, but often much shorter on Railway's managed MySQL). When the connection drops mid-job, all subsequent DB writes ‚Äî including the `status = "completed"` update and the `outputFileUrl` write ‚Äî fail silently or throw, causing the job to appear stuck or to complete with no output file.

**Root Cause 2C ‚Äî The `firmCount` stored in the DB may not match the actual parsed firm count.**

When `confirmAndStart` is called, the `firmCount` passed to `createEnrichmentJob` comes from the client-side preview (`previewData.firmCount`), which is computed during the upload/preview step. If the user's Excel file has rows that are later filtered out by `parseInputExcel` (e.g., rows with missing URLs, email addresses in the URL column, or duplicate entries), the actual number of firms processed will be less than `firmCount`. The UI progress bar shows `processedCount / firmCount`, so if `firmCount = 100` but only 60 rows were valid, the bar will never reach 100% even after all valid rows are processed ‚Äî and the job may appear "stuck" at 60%.

### Recommended Fixes

| Fix | Location | Effort |
|---|---|---|
| Add `keepAlive.start()` to `processAgentJob` | `routers.ts` line ~931 | Trivial (1 line) |
| Remove the direct `processEnrichmentJob()` call from `resumeJob` ‚Äî set status to `pending` and let the worker pick it up | `routers.ts` ‚Äî `resumeJob` mutation | Low |
| After `parseInputExcel`, update `firmCount` in the DB to reflect the actual parsed count | `routers.ts` ‚Äî `processAgentJob` | Low |

## Issue 3: Cancel/Stop Does Not Actually Stop the Running Job

### Symptom

Clicking "Cancel" in the UI shows a success toast and the job card updates to "cancelled" status. However, the Railway worker logs continue to show the job processing URLs for minutes or even hours after cancellation.

### Root Causes

**Root Cause 3A ‚Äî The `cancelJob` mutation only writes to the database. It does not signal the worker process.**

The `cancelJob` mutation in `routers.ts` executes a single SQL `UPDATE` to set `status = "cancelled"`. It does not call `markJobCancelled()`, does not send a signal to the worker process, and does not interact with the in-memory cancellation flag in any way:

```typescript
// cancelJob mutation ‚Äî only writes to DB
await db.update(enrichmentJobs)
  .set({ status: "cancelled", completedAt: new Date() })
  .where(...);
return { success: true };  ‚Üê returns immediately, worker still running
```

The worker's `checkForCancellation` timer polls the DB every 5 seconds and calls `markJobCancelled()` when it sees `status = "cancelled"`. This means there is a **5-second detection window** after the user clicks Cancel before the worker even knows about it.

**Root Cause 3B ‚Äî The cancellation check only fires between firms, not during in-flight LLM calls or network fetches.**

The `isCancelled` check in `agentScraper.ts` is called at 5 specific points:

1. Before fetching the URL (before Jina/Puppeteer call)
2. After the fetch, before page classification
3. Before the initial `extractProfileFields` call
4. At the start of each hop loop iteration
5. (Implicitly) when `JOB_CANCELLED` is thrown and caught in the firm loop

None of these checks interrupt an **in-progress LLM call**. If the worker is currently inside `queuedLLMCall()` ‚Äî which can take 2‚Äì15 seconds per call ‚Äî the cancellation flag is not checked until that call completes and control returns to the calling function. With `CONCURRENCY = 50`, there can be up to 50 simultaneous LLM calls in flight at the moment of cancellation. Each of those calls must complete before the cancellation flag is checked for that firm. For a firm that requires 5 hops √ó 3 LLM calls per hop, the worker may continue processing for up to 75 LLM calls after cancellation is detected ‚Äî potentially several minutes of additional work.

**Root Cause 3C ‚Äî The stale job recovery logic can accidentally re-queue a cancelled job.**

The `findNextJob` function in `worker.ts` checks for stale jobs (status = "processing" with an old heartbeat) and resets them to "pending". However, the stale job query does not exclude jobs that were cancelled:

```typescript
// worker.ts ‚Äî findNextJob stale recovery
const staleJobs = await db.select()
  .from(enrichmentJobs)
  .where(
    and(
      eq(enrichmentJobs.status, "processing"),  ‚Üê only checks "processing"
      or(lt(heartbeatAt, staleThreshold), isNull(heartbeatAt))
    )
  );
```

In a race condition where: (1) the user cancels the job, (2) the `cancelJob` mutation sets `status = "cancelled"`, (3) the worker has not yet detected the cancellation and is still sending heartbeats, and (4) the Railway container restarts ‚Äî the job's status in the DB will be "cancelled" and the stale recovery will not pick it up (correct). However, if the cancellation write races with a heartbeat write and the heartbeat wins, the status may briefly revert to "processing" with a fresh heartbeat, making the job appear active again. This is a low-probability race condition but has been observed in practice.

**Root Cause 3D ‚Äî The `firmQueue.length = 0` cancellation shortcut only drains the queue; it does not abort the 50 concurrent workers already running.**

When cancellation is detected in the firm loop, the code sets `firmQueue.length = 0` to prevent new firms from being picked up. But `Promise.allSettled` is already running 50 concurrent `runWorker` coroutines. Each of those coroutines is in the middle of `await scrapeUrl(...)`. Setting `firmQueue.length = 0` means no new firms will start, but all 50 currently-running scrapes will complete before the `Promise.allSettled` resolves. This is why the Railway logs show continued activity for minutes after cancellation.

### Recommended Fixes

| Fix | Location | Effort |
|---|---|---|
| Use an `AbortController` per job; pass the `AbortSignal` into `fetch()` calls inside Jina and Puppeteer fetchers so in-flight network requests are immediately aborted | `jinaFetcher.ts`, `scraper/ComprehensiveScraper.ts`, `agentScraper.ts` | Medium |
| Add `isCancelled` checks inside the hop loop's sub-page fetch and sub-page extraction calls | `agentScraper.ts` ‚Äî profile hop loop | Low |
| Reduce `CONCURRENCY` from 50 to a more conservative value (e.g., 10‚Äì15) to reduce the blast radius of a cancel event | `routers.ts` | Trivial |
| Add a guard in `findNextJob` to exclude jobs with `status = "cancelled"` from stale recovery | `worker.ts` | Trivial (1 line) |

## Should a Major Overhaul Be Considered?

The existing architecture is fundamentally sound. The worker/web-server split, the DB-based job queue, the heartbeat system, and the 4-level scraping fallback are all well-designed. The bugs are concentrated in three narrow areas: a missing one-liner (`keepAlive.start()`), a content truncation limit, and the absence of `AbortController` propagation into network calls.

**A full overhaul is not recommended.** However, two architectural improvements are worth planning for a future iteration:

1. **Replace the in-memory `cancelledJobs` Set with a Redis pub/sub or a DB-polled abort signal that is wired directly into the `fetch()` calls.** This would make cancellation near-instantaneous regardless of what the worker is doing.

2. **Move `processAgentJob` and `processEnrichmentJob` entirely out of `routers.ts` and into `worker.ts`.** Currently, these functions are exported from `routers.ts` and imported into `worker.ts`, which means the web server module is loaded into the worker process (and vice versa for the `resumeJob` path). Separating these into a dedicated `jobProcessor.ts` module would eliminate the dual-process confusion and make it impossible to accidentally call a job processor from the web server.

## Prioritized Fix Order

Given the severity and effort ratings, the recommended implementation order is:

**Phase 1 ‚Äî Quick Wins (< 1 hour total):**
1. Add `keepAlive.start()` to `processAgentJob` ‚Äî fixes the DB connection drop that causes silent failures.
2. Add the `cancelled` status exclusion to the stale job recovery query in `worker.ts`.
3. Reduce `CONCURRENCY` from 50 to 10 in `processAgentJob`.
4. Fix `resumeJob` to set status `pending` instead of calling `processEnrichmentJob()` directly.

**Phase 2 ‚Äî Core Fixes (2‚Äì4 hours total):**
5. Raise the section cap from 8 to 15 in `generateExtractionPlan` and update the UI.
6. Add `isCancelled` checks inside the sub-page hop loop (after each sub-page fetch and after each sub-page extraction).
7. Update `firmCount` in the DB after `parseInputExcel` to reflect the actual parsed count.
8. Fix `mergeResults` to prefer the longer value.

**Phase 3 ‚Äî Structural Improvements (1‚Äì2 days):**
9. Implement chunked content extraction in `extractProfileFields` to handle pages > 25K characters.
10. Wire `AbortController` through the Jina fetcher and Puppeteer scraper for instant cancellation of in-flight requests.
11. Add a post-generation key-matching validation in `generateExtractionPlan`.
